## 1. Executive Summary

The field of Automatic Item Generation (AIG) has undergone a paradigm shift in 2024-2025 with LLMs. Key findings:

- **LLM-generated items are now comparable to human-authored items** in quality (Kowal et al. 2025, Bulut et al. 2024), with 10x faster generation time.
- **Pre-calibration of IRT parameters without human data is possible** using AutoIRT (Duolingo, 2024) — NLP features predict difficulty/discrimination before any test-taker touches the item.
- **O*NET (974 occupations) and ESCO (3,008 occupations) provide free APIs** with structured competency data — the exact input our pipeline needs.
- **Google Vantage (April 2026)** validates the approach of using an "Executive LLM" to steer adaptive assessment — their AI evaluator matched human inter-rater reliability.
- **Minimum viable pipeline is buildable in 2-3 weeks** with our existing stack. Full psychometric validation requires 3-6 months of data collection.

**The critical insight:** VOLAURA already has the hardest parts built (3PL IRT engine, BARS evaluator, anti-gaming quality gates). What's missing is the item generation pipeline and the role-to-competency mapping layer.

## 3. Psychometric Standards for Generated Items

### ISO 10667 (Assessment Service Delivery)

ISO 10667-1:2020 and ISO 10667-2:2020 establish requirements for assessment in work and organizational settings.

**Key requirements for our pipeline:**
- **Validity evidence** must be documented for every assessment purpose (ISO 10667-2, Clause 5.3)
- **Reliability** must be quantified — our SE threshold stopping rule satisfies this
- **Fairness** across demographic groups must be monitored (Clause 5.5)
- **Transparency** — assessment participants have the right to understand how they were assessed (our DeCE framework addresses this)
- **Competence of assessors** — extends to AI systems acting as assessors

**VOLAURA compliance status:** DeCE framework already cites ISO 10667-2 Clause 6.7. Dynamic items don't break compliance as long as we maintain validity evidence and fairness monitoring.

### APA/AERA/NCME Standards for Educational and Psychological Testing (2014)

The "Bible" of psychometrics. Key standards relevant to generated items:

- **Standard 1.0 (Validity):** "Clear articulation of each intended test score interpretation for each specified use." — We need validity evidence per competency domain, not per item.
- **Standard 3.0 (Fairness):** "All steps in the testing process, including test design, validation, development, administration, and scoring procedures, should be designed in such a manner as to minimize construct-irrelevant variance."
- **Standard 4.0 (Test Design):** When items are generated algorithmically, the generation process itself becomes the unit of validation — you validate the item model (template + constraints), not each individual item.
- **Standard 4.10:** "When a test is to be scored by computer, the test manual should provide evidence of the accuracy and consistency of the scoring."

**Critical insight for VOLAURA:** The Standards explicitly address automated item generation. The requirement is to validate the **item generation model** (the prompt template + constraints + quality gate), then individual items inherit validity from the model. This is far more scalable than validating each generated item.

### ITC Guidelines on Computer-Based and Internet-Delivered Testing

Key guidelines:
- **Guideline 3:** Technology should be appropriate for the construct being measured
- **Guideline 4:** Steps should be taken to ensure security and integrity
- **Guideline 7:** Test-taker data must be protected

**What makes a generated question "valid" vs garbage:**

1. **Construct alignment** — The question measures what it claims to measure (the named competency), not adjacent constructs
2. **Appropriate difficulty** — Maps to the intended position on the ability scale
3. **Discrimination** — Differentiates between high and low ability test-takers
4. **Fairness** — No systematic advantage/disadvantage for demographic groups
5. **Answerability** — Has a defensible correct/best response that subject matter experts agree on
6. **Gaming resistance** — Cannot be answered correctly by pattern-matching alone (our GRS gate)
7. **Domain accuracy** — All factual claims in the question are correct (critical for LLM-generated content)

## 4. Existing Platforms & Competitors

### Tier 1: Enterprise Assessment Platforms

#### Pymetrics (now Harver)
- **Approach:** Game-based behavioral assessments using neuroscience, NOT knowledge testing
- **Scale:** 100M+ assessments completed
- **Key insight:** Non-verbal, gamified exercises bypass cultural/language bias entirely
- **Weakness for our use case:** Measures cognitive traits, not professional competency knowledge
- **What to learn:** Their 98% completion rate proves engagement matters. Our 5-minute target is correct.

#### Criteria Corp
- **Approach:** Traditional I-O psychology assessments + AI-powered interview scoring
- **Innovation:** Interview Intelligence (IIQ) scores video interviews with I-O psychologist-level accuracy
- **What to learn:** They emphasize that AI is "not the right tool for every client" — measured approach to AI claims

#### Arctic Shores
- **Approach:** Task-based gamified assessment using 12,000 data points per candidate
- **Key claim:** "ChatGPT-proof" — tasks require behavioral demonstration, not knowledge recall
- **What to learn:** Explicitly designed to resist LLM-assisted cheating. Our anti-gaming gates serve the same purpose.

#### Questionmark
- **Approach:** Enterprise assessment with AI-assisted item authoring and scoring
- **Technical:** Built on Learnosity. AI helps create items but doesn't generate them autonomously.
- **What to learn:** Their AI scoring for open-ended items is similar to our BARS approach.

#### Kryterion
- **Approach:** Professional certification testing with AI-enhanced proctoring
- **Innovation:** Using GenAI for dynamic item selection and simulated performance assessments
- **Key quote:** "GenAI can dynamically select future questions based on real-time candidate performance analysis"
- **What to learn:** They see adaptive + generative as the future but are moving cautiously due to certification liability.

### Tier 2: AI-Native Assessment

#### Assessment Systems (Assess.com)
- **Most relevant competitor.** Direct LLM support for item generation with iterative prompt engineering.
- **Offers:** AI-powered adaptive and multistage testing
- **CAT implementation:** Full IRT-based CAT, documented publicly
- **What to learn:** They allow users to specify topics and AI generates draft items for review. Human-in-the-loop model.

#### Workera
- **Approach:** AI-powered skills intelligence for enterprise workforces
- **Key differentiator:** Measures technical, soft, and cognitive skills using real-world task formats
- **What to learn:** Their "science-backed, adaptive" positioning is closest to VOLAURA's value proposition

#### Google Vantage (Research, April 2026)
- **Approach:** Executive LLM steers AI avatars to create adaptive assessment scenarios
- **Validation:** AI evaluator matched human inter-rater reliability (n=188, NYU partnership)
- **Key innovation:** Single coordinating LLM generates all participant responses, enabling pedagogical control
- **What to learn:** Validates that LLM-as-evaluator can match human expert reliability. Our BARS approach is validated by this research.

## 5. Technical Approaches to Item Generation

### 5.1 Academic AIG Landscape (2024-2025)

The field has produced several frameworks:

#### LM-AIG: Multi-Agent AIG System (Springer, 2025)
- Uses AutoGen to orchestrate multiple LLM agents for item development
- Agents handle: generation, content review, linguistic evaluation, bias assessment, revision
- **Directly applicable:** Maps to our existing swarm architecture

#### Human-in-the-Loop (HITL) Framework (MDPI, 2025)
- AI accelerates generation; human expert retains authority over selection, calibration, bias review
- **Key finding:** "Human oversight remains critical, particularly for content validation, bias detection, and evaluation of item accuracy" (Kowal et al. 2025)

#### LLM Temperature for AIG
- Temperature 0.7-1.0 recommended for diverse item generation (Li et al. 2024)
- Lower temperatures produce more similar items; higher temperatures risk incoherence
- **Recommendation for VOLAURA:** Use temperature 0.8 for initial generation, 0.2 for evaluation

#### LLM vs Human Quality Comparison
- "Time taken to generate MCQs by humans was about ten times longer than by ChatGPT, while the quality of the questions was comparable" (Bulut et al. 2024)
- Large-scale field study (arXiv 2025): LLM-generated and human items had "similar item difficulty distributions in an IRT analysis"

### 5.2 Item Generation Architecture for VOLAURA

Recommended multi-stage pipeline:

```
Stage 1: Domain Knowledge Assembly (RAG)
  Input: Role title, experience level, competency framework
  Action: Retrieve domain-specific knowledge from O*NET/ESCO + curated sources
  Output: Domain knowledge context (standards, regulations, best practices)

Stage 2: Item Template Selection
  Input: Competency to assess, question type (scenario, situational judgment, technical)
  Action: Select from validated item templates (see below)
  Output: Structured prompt with template + domain context

Stage 3: Item Generation (Gemini 2.5 Flash)
  Input: Structured prompt
  Action: Generate question text + expected concepts + keywords + BARS rubric
  Output: Raw question dict matching our DB schema

Stage 4: Quality Validation (Automated)
  Action: Run through existing quality_gate.py (10-point checklist + GRS + adversarial gate)
  If fails: regenerate with feedback on failures (max 3 attempts)
  Output: Validated question dict

Stage 5: IRT Pre-Calibration
  Action: Predict irt_a, irt_b, irt_c using AutoIRT-style feature extraction
  Output: Question with estimated IRT parameters

Stage 6: Domain Fact-Check (RAG Verification)
  Action: Verify all factual claims against authoritative sources
  Output: Verified question or flagged for human review
```

### 5.3 Item Templates (Validated Structures)

Based on the literature, these template types produce the highest quality generated items:

**Template 1: Situational Judgment (Best for professional competencies)**
```
You are a [ROLE] with [EXPERIENCE_LEVEL] years of experience.
[SCENARIO_DESCRIPTION — 2-3 sentences describing a realistic workplace situation]
Describe step by step what you would do and why.
```

**Template 2: Critical Incident**
```
A colleague in your [DEPARTMENT] has [INCIDENT_DESCRIPTION].
This has resulted in [CONSEQUENCE].
Explain how you would handle this situation, considering [COMPETENCY_AREA].
```

**Template 3: Technical Knowledge Application**
```
You are reviewing [ARTIFACT/PROCESS] in your role as a [ROLE].
You notice [OBSERVATION — specific technical detail].
What actions would you take? Explain your reasoning.
```

**Template 4: Stakeholder Communication**
```
[STAKEHOLDER_TYPE] has raised [CONCERN/REQUEST].
As a [ROLE], draft your response addressing their concern
while maintaining [STANDARD/REGULATION] compliance.
```

### 5.4 Prompt Engineering for Quality Items

Based on the research, the generation prompt must include:

```python
ITEM_GENERATION_PROMPT = """
You are an expert psychometrician generating assessment questions.

DOMAIN CONTEXT:
{domain_knowledge}  # From O*NET/ESCO + RAG

COMPETENCY TO ASSESS: {competency_name}
COMPETENCY DEFINITION: {competency_definition}
TARGET DIFFICULTY: {difficulty_level}  # easy/medium/hard
ROLE: {role_title}
EXPERIENCE LEVEL: {experience_years} years

REQUIREMENTS:
1. Write a scenario-based question (40-80 words) that requires
   the test-taker to DEMONSTRATE the competency, not just name it
2. The scenario must be realistic for someone in this role
3. Include 2-4 expected concepts, each with:
   - name (snake_case identifier)
   - weight (sum to 1.0)
   - keywords (3-5 multi-word phrases per concept)
4. Keywords must NOT appear in the question text
5. The question must contain a narrative trigger
   (describe, explain, how would you, what would you)
6. All factual references (regulations, standards, procedures)
   must be real and current

DIFFICULTY CALIBRATION:
- Easy (irt_b ~ -1.0): Common scenario, straightforward response expected
- Medium (irt_b ~ 0.0): Nuanced scenario, competing priorities
- Hard (irt_b ~ 1.0): Complex multi-stakeholder scenario, ethical dilemma

ANTI-PATTERNS TO AVOID:
- Questions answerable by listing keywords
- Questions with culturally specific assumptions
- Trick questions or gotcha scenarios
- Questions requiring knowledge not typical for the role
- Questions with a single "right" answer (we assess approach, not memorization)

OUTPUT FORMAT (JSON):
{
  "scenario_en": "...",
  "scenario_az": "...",
  "competency_slug": "...",
  "type": "open_ended",
  "expected_concepts": [
    {
      "name": "concept_name",
      "weight": 0.5,
      "keywords": ["multi-word phrase 1", "multi-word phrase 2", "phrase 3"]
    }
  ],
  "irt_a": 1.5,  // estimated discrimination
  "irt_b": 0.0,  // estimated difficulty
  "irt_c": 0.0,  // guessing parameter (0 for open-ended)
  "difficulty_rationale": "Why this difficulty level",
  "domain_references": ["Source 1", "Source 2"]
}
"""
```

## 6. IRT Pre-Calibration for New Items

### 6.1 The Cold-Start Problem

Generated items have never been administered to humans. Traditional IRT calibration requires:
- **1PL (Rasch):** 100-250 test-takers minimum
- **2PL:** 500 test-takers minimum
- **3PL:** 1,000+ test-takers minimum

This is a chicken-and-egg problem: we need IRT parameters to run CAT, but we need CAT responses to calibrate IRT parameters.

### 6.2 AutoIRT: The Solution (Duolingo, 2024)

AutoIRT uses machine learning to predict IRT parameters from item features, without any response data.

**How it works:**
1. Extract NLP features from the question text (BERT embeddings, linguistic features)
2. Train an AutoML model (Random Forest + LightGBM + XGBoost ensemble via AutoGluon)
3. Two-stage MCEM algorithm:
   - Stage 1: Non-parametric AutoML classifier predicts response probabilities
   - Stage 2: Parametric IRT model fitted using predicted probabilities
4. Result: Estimated irt_a, irt_b, irt_c before any human takes the test

**Performance:** Retest reliability and item calibration correlations exceeding 0.98 in warm-start conditions. Substantial gains even in cold-start.

### 6.3 Recommended Pre-Calibration Strategy for VOLAURA

#### Phase 1: LLM-Based Estimation (Day 1 — No Training Data)

Use the LLM itself to estimate difficulty based on prompt:

```python
CALIBRATION_PROMPT = """
You are an IRT psychometrician. Estimate the 3PL parameters for this question.

Question: {question_text}
Target role: {role} with {experience} years experience
Competency: {competency}

Estimate:
- irt_a (discrimination, 0.3-3.0): How well does this question differentiate
  between high and low ability? Higher = more discriminating.
- irt_b (difficulty, -4.0 to 4.0): Where on the ability scale does this question
  sit? -2 = very easy, 0 = medium, +2 = very hard.
- irt_c (guessing, 0.0-0.35): Can someone score well by guessing?
  0.0 for open-ended, 0.2-0.25 for 4-option MCQ.

Consider:
- Scenario complexity (number of competing factors)
- Required knowledge depth (surface vs specialized)
- Response sophistication expected
- Typical experience level needed
"""
```

**Expected accuracy:** Rough (+/- 0.5 on difficulty scale). Sufficient for initial CAT but not for high-stakes scoring.

#### Phase 2: Feature-Based Prediction (Week 2 — With Training Data from Static Bank)

Train a simple model on our existing 123 questions + their observed IRT parameters:

```python
Features to extract per question:
- word_count: len(scenario_en.split())
- sentence_count: number of sentences
- flesch_kincaid_grade: readability score
- concept_count: len(expected_concepts)
- total_keyword_count: sum of all keywords
- multi_word_keyword_ratio: fraction of multi-word keywords
- narrative_trigger_count: count of trigger words
- question_type: open_ended vs mcq
- competency_category: broad domain classification
- grs_score: Gaming Resistance Score
- bert_embedding: 768-dim from Gemini text-embedding-004
```

Train a LightGBM/XGBoost model to predict (irt_a, irt_b, irt_c) from these features. With 123 training items, even simple models should improve over pure LLM estimation.

#### Phase 3: Online Calibration (Month 2+ — With Live Data)

As test-takers complete assessments with generated items:
1. Embed 1-2 "pilot" generated items per session alongside calibrated items
2. Use responses to update IRT parameters via online calibration (Elo-based or Bayesian)
3. After 50-100 responses per item, parameters stabilize
4. Items graduate from "pilot" to "operational" status

#### Phase 4: Full AutoIRT (Month 6+ — With Sufficient Data)

With 500+ calibrated items and thousands of responses, train a full AutoIRT model:
- AutoGluon ensemble on tabular features
- BERT/Gemini embeddings for semantic features
- Predicts parameters for any new item with high accuracy
- Cold-start items immediately usable in CAT

### 6.4 Minimum Sample Sizes (Literature Consensus)

| Scenario | Sample Size Needed | Timeline at 100 users/day |
|----------|--------------------|--------------------------|
| Rough difficulty estimate (LLM) | 0 (pre-calibration) | Instant |
| Bayesian update from responses | 50-100 per item | 1-2 weeks per item |
| Stable 2PL calibration | 500 per item | 5 weeks per item |
| Stable 3PL calibration | 1,000 per item | 10 weeks per item |
| Full AutoIRT training set | 500+ calibrated items | 3-6 months |

**Practical recommendation:** Start with LLM estimation + 1PL Rasch (simplest model, needs fewest responses). Upgrade to 2PL/3PL as data accumulates. For open-ended questions, c=0 always (no guessing on open-ended), so we effectively need 2PL calibration only.

## 7. Competency Frameworks & Role Mapping

### 7.1 O*NET (US Department of Labor)

**Coverage:** 923 data-level occupations across the US economy
**Data per occupation:**
- Tasks (what people do)
- Knowledge areas (what people need to know)
- Skills (developed capacities)
- Abilities (enduring attributes)
- Work Activities (general types of job behaviors)
- Work Context (physical, social, organizational conditions)
- Technology Skills (tools and technologies used)
- Education requirements
- Related occupations

**API Access:**
- REST API: `services.onetcenter.org/v1.9/`
- Free with registration
- Returns JSON
- Supports search by keyword, title, O*NET-SOC code
- Skills ranked by importance (1-5 scale) and level (1-7 scale)

**Example API call:**
```
GET https://services.onetcenter.org/v1.9/online/occupations/35-1011.00
→ Returns: Chefs and Head Cooks with all competency data
```

**Competency data available per occupation:**
- 35 Knowledge areas with importance/level ratings
- 35 Skills with importance/level ratings
- 52 Abilities with importance/level ratings
- Hundreds of tasks specific to the occupation
- Technology skills (specific tools)

### 7.2 ESCO (European Commission)

**Coverage:** 3,008 occupations, 13,890 skills/competences, 27 languages
**Data per occupation:**
- Essential skills and competences
- Optional skills and competences
- Essential knowledge
- Optional knowledge
- Related occupations (via ISCO-08)

**API Access:**
- REST API: `ec.europa.eu/esco/api`
- Free, no registration required
- Returns JSON-LD (linked data)
- Full text search across all languages
- Downloadable as CSV, Excel, RDF, JSON

**Key advantage over O*NET:** Multilingual (27 languages including close-to-Azerbaijani Turkish). Better for our AZ-primary user base.

### 7.3 SFIA (IT-Specific)

**Coverage:** 100+ IT skills across 7 levels of responsibility
**Format:** Available as JSON, RDF Turtle, Excel, PDF
**Access:** Free with registration
**Best for:** IT professional assessment specifically
**Latest version:** SFIA 9 (2025)

### 7.4 Role Mapping Pipeline

The automatic mapping flow:

```
User Input: "Chef, 5 years, Michelin restaurant"
                    ↓
Step 1: Role Normalization (LLM)
  → Standardized: "Chef / Head Cook"
  → Experience: Senior (5+ years)
  → Context: Fine dining, Michelin-level
                    ↓
Step 2: O*NET Mapping
  → API search: "chef head cook"
  → Match: 35-1011.00 (Chefs and Head Cooks)
  → Retrieve: 35 knowledge areas, 35 skills, tasks
                    ↓
Step 3: ESCO Cross-Reference
  → Map O*NET code to ESCO via ISCO-08
  → Retrieve: Essential/optional skills in 27 languages
  → Get AZ translations where available
                    ↓
Step 4: Competency Selection (LLM)
  → Input: All O*NET/ESCO competencies for this role
  → Filter by: Experience level, context (Michelin)
  → Select: Top 5-8 competencies most critical for this role+level
  → Map to VOLAURA AURA categories where possible
                    ↓
Step 5: Competency Enrichment
  → For each selected competency:
    - Industry standards (food safety: HACCP, ServSafe)
    - Regulatory requirements (local health codes)
    - Professional best practices
    - Level-appropriate expectations
                    ↓
Output: Structured competency profile for item generation
```

```
│  3. Present question to user                                     │
│  4. Score response via BARS (Gemini → Groq → OpenAI → keyword)  │
│  5. Update theta via EAP                                         │
│  6. Check stopping rule (SE <= threshold or max items)           │
│  7. Repeat from 2 until stopped                                  │
│  Output: theta per competency                                    │
└───────────────────────────┬─────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────────┐
│  SCORING & PROFILE                                               │
│  1. theta → 0-100 per competency (theta_to_score)               │
│  2. Weighted AURA score (role-specific weights)                  │
│  3. Badge assignment (Platinum/Gold/Silver/Bronze)               │
│  4. Competency breakdown with DeCE explanations                  │
│  5. Public profile generation                                    │
│  Output: AURAScore + Badge + PublicProfile                       │
└─────────────────────────────────────────────────────────────────┘
```

### 8.2 Latency Budget

Target: 10-question test in 5 minutes. System overhead must be minimal.

| Operation | Budget | Approach |
|-----------|--------|----------|
| Role mapping + competency extraction | 2-3s (once) | O*NET API call + LLM parse |
| Question generation (15-20 items) | 10-15s (once, pre-session) | Parallel Gemini calls |
| Quality gate validation (per item) | <100ms | CPU-only, no LLM |
| IRT pre-calibration (per item) | <50ms | Feature extraction + model inference |
| BARS evaluation (per response) | 2-5s | Gemini 2.5 Flash |
| CAT item selection | <10ms | MFI calculation |
| EAP theta update | <10ms | Quadrature computation |

**Total system overhead per question:** ~3-6 seconds (dominated by BARS evaluation)
**Pre-session setup:** ~15-20 seconds (generation + validation)

This is well within the 5-minute target for 10 questions.

## 9. Risks & Failure Modes

### 9.1 LLM Hallucination in Domain Knowledge

**Risk:** LLM generates questions referencing non-existent regulations, standards, or procedures.
**Example:** Asking a chef about "FDA Code 2024 Section 7.3.2" when no such section exists.
**Severity:** HIGH — destroys credibility with professionals who know their domain.

**Mitigations:**
1. **RAG grounding:** All domain knowledge retrieved from authoritative sources (O*NET, ESCO, government databases), not from LLM's parametric knowledge
2. **Fact verification step:** Second LLM call with explicit instruction to verify all factual claims against provided sources
3. **No specific regulation citations:** Generate scenario-based questions that test understanding of principles, not recall of specific codes
4. **Domain expert review queue:** Flag questions for human review when confidence is low
5. **Chain-of-Verification (CoVe):** LLM drafts question, generates verification queries, answers them independently, revises

### 9.2 Cultural Bias

**Risk:** Generated questions assume Western workplace norms, legal frameworks, or cultural practices.
**Example:** Asking about "open-door management policy" in cultures where hierarchy is strict and questioning superiors is inappropriate.
**Severity:** HIGH — VOLAURA targets Azerbaijan (AZ-primary) + international.

**Mitigations:**
1. **Culture-aware generation prompt:** Include locale and cultural context in the generation prompt
2. **Universal scenarios:** Focus on competency demonstration through universal workplace situations (team coordination, problem-solving, communication)
3. **Avoid culture-specific references:** No idioms, no culture-specific business practices, no assumptions about workplace structure
4. **AZ-specific review:** Generated AZ translations reviewed against cultural norms
5. **Bias detection in responses:** Monitor for systematic score differences across demographic groups

### 9.3 Gaming & Cheating

**Risk:** When each user gets unique questions, traditional cheating (sharing answers) is harder, but new risks emerge:
- Using LLMs to answer open-ended questions
- Sharing the type of questions asked for a role
- Reverse-engineering the scoring rubric from DeCE feedback

**Mitigations (already built):**
1. **BARS anti-injection:** System prompt prevents gaming via meta-instructions
2. **Keyword stuffing detection:** 4-gate anti-gaming in `_keyword_fallback()`
3. **Epsilon-greedy item selection:** Prevents hot-item exposure
4. **GRS quality gate:** Ensures questions resist pattern-matching

**Additional mitigations needed:**
5. **Response time anomaly detection:** LLM-assisted answers tend to be too fast (paste) or too slow (typing from another window)
6. **Linguistic fingerprinting:** Detect LLM-generated responses via perplexity analysis
7. **Consistency checking:** Cross-reference competency scores with self-reported experience (a "5-year chef" scoring Bronze on food safety is suspicious)

### 9.4 Legal Liability

**Risk:** AI-generated assessments used for employment decisions face legal scrutiny.
**Severity:** MEDIUM-HIGH — depends on jurisdiction and use case.

**Key regulatory frameworks:**
- **EU AI Act (Aug 2025):** AI systems used in employment assessment are classified as "high-risk" (Annex III, para 4). Requires: risk management system, data governance, technical documentation, transparency, human oversight.
- **US EEOC Guidelines:** Title VII requires that employment tests demonstrate validity and do not disproportionately impact protected groups.
- **ISO 10667:** Provides defensible framework for assessment service delivery.

**Mitigations:**
1. **VOLAURA is not an employment test** — it's a voluntary competency assessment for professional development. This significantly reduces legal exposure compared to hiring tools.
2. **Transparency:** DeCE framework provides full explainability
3. **No adverse impact claims:** Monitor group-level score distributions
4. **Disclaimer:** "AURA scores reflect demonstrated competency in a timed assessment and are not a substitute for professional certification"
5. **ISO 10667 alignment:** Already partially implemented

### 9.5 Psychometric Validity of Generated Items

**Risk:** Generated items may not actually measure the intended competency (low construct validity).
**Severity:** HIGH — the entire platform's credibility depends on this.

**Mitigations:**
1. **Template-based generation:** Validated item templates constrain the space of possible outputs
2. **Quality gate:** 10-point checklist catches structural problems
3. **Convergent validity testing:** Compare scores from generated items vs calibrated items for the same competency
4. **Field testing protocol:** Embed pilot items alongside operational items, compare item statistics
5. **Expert review for high-stakes domains:** Medical, legal, financial assessment items require SME sign-off

### 9.6 Over-Reliance on LLM Evaluation

**Risk:** BARS evaluator may have systematic biases (e.g., favoring verbose answers, penalizing non-native English speakers).
**Severity:** MEDIUM — already mitigated by multi-provider fallback.

**Mitigations:**
1. **Multi-model consensus:** Gemini + Groq + OpenAI provide cross-model validation
2. **Keyword fallback as floor:** Ensures scoring continues even when all LLMs fail
3. **Calibration against human raters:** Periodic comparison of BARS scores vs human SME ratings
4. **Language accommodation:** Explicit prompt to evaluate content quality regardless of grammar/fluency

## 10. Implementation Roadmap

### Phase 1: MVP (Weeks 1-2) — "Smart Question Generation"

**Goal:** Generate competency-specific questions for any stated role using LLM + O*NET

**Deliverables:**
1. `RoleMapper` service: Parse user input → O*NET API → competency profile
2. `ItemGenerator` service: Competency profile → Gemini prompt → question + BARS rubric
3. Integration with existing `quality_gate.py` for validation
4. LLM-based IRT pre-calibration (rough estimates)
5. New API endpoint: `POST /assessment/generate-session`
6. Simple caching of generated questions by O*NET code

**Scope:**
- Start with 10 most common occupations
- Use fixed AURA weights (current 8 competencies) mapped to role
- No online calibration yet
- Flag all items as "pilot" status

**Architecture:**
```python
# New files
apps/api/app/services/role_mapper.py       # O*NET API + competency extraction

apps/api/app/services/irt_calibrator.py    # Pre-calibration service
apps/api/app/routers/dynamic_assessment.py # New endpoints
```

**Risk level:** LOW — uses existing engine, adds generation layer on top

### Phase 2: Calibration Pipeline (Weeks 3-6) — "Learning from Usage"

**Goal:** Refine IRT parameters from actual test-taker responses

**Deliverables:**
1. Online calibration system (Elo-based parameter updates)
2. Pilot/operational item status tracking
3. Feature extraction pipeline for AutoIRT training
4. Item exposure monitoring dashboard
5. A/B testing: generated vs static items comparison

**Key metric:** Correlation between pre-calibrated and empirically calibrated IRT parameters

### Phase 3: Domain Expansion (Weeks 7-12) — "Any Profession"

**Goal:** Scale to 100+ occupations with verified domain knowledge

**Deliverables:**
1. ESCO integration for multilingual competency data
2. RAG pipeline with domain-specific knowledge bases
3. Automated fact-checking for generated questions
4. Cultural adaptation layer for AZ/TR/RU locales
5. Role-specific AURA weight profiles
6. SFIA integration for IT roles specifically

### Phase 4: Full AutoIRT (Months 4-6) — "Self-Improving Engine"

**Goal:** System that predicts IRT parameters accurately for any new item

**Deliverables:**
1. Full AutoIRT model trained on 500+ calibrated items
2. Cold-start items immediately usable in CAT
3. Automated item retirement (low-performing items rotated out)
4. Psychometric quality dashboard
5. Bias detection and fairness monitoring
