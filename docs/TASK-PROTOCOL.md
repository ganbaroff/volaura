# TASK PROTOCOL v1.0 — Swarm Critique Loop

**Версия:** 1.0 | **Добавлено:** 2026-03-28
**Почему существует:** Claude самостоятельно проверяет свою работу — это не верификация, это самообман.
Рой — внешняя проверка, которую нельзя подделать.

---

## Как использовать

Yusif отправляет этот файл перед каждой задачей с командой: **"Загрузи TASK-PROTOCOL.md и начни."**
Claude читает, объявляет: `"TASK-PROTOCOL v1.0 loaded."` и следует шагам.

**Главное правило:** Каждый шаг производит **артефакт** с именем.
Нельзя перейти к шагу N без артефакта шага N-1.
"Понятно", "принято", "учёл" — не артефакты. Это обход.

---

## ШАГИ

### ШАГ 0 — СКИЛЫ (артефакт: `Skills: [список]`)

1. Открыть CLAUDE.md → раздел Skills Matrix
2. Выписать ВСЕ строки, которые подходят к задаче (не одну, не "достаточно")
3. Для каждого скила: 2 конкретных правила для ЭТОЙ задачи
4. Написать: `"Skills: [список]. Правила для задачи: [буллеты]"`

→ БЕЗ ЭТОЙ ДЕКЛАРАЦИИ — шаг 1 не начинается.

---

### ШАГ 1 — SCOPE LOCK (артефакт: 4 строки)

```
IN:      [что конкретно доставляется — тестируемый результат]
NOT IN:  [что явно откладывается]
SUCCESS: [как узнаём что готово — не "код написан", а наблюдаемый результат]
TOKENS:  [оценка бюджета]
```

→ БЕЗ ВСЕХ 4 СТРОК — шаг 2 не начинается.

---

### ШАГ 2 — ПЛАН (артефакт: нумерованные шаги)

Написать план реализации: пронумерованные шаги, каждый ≤ 2 строки.
Не писать код. Не открывать файлы для изменения.

→ ПЛАН ГОТОВ — отправить рою (шаг 3).

---

### ШАГ 3 — РОЙ КРИТИКУЕТ ПЛАН (артефакт: critique-round-1.md)

Запустить Agent (haiku) с промптом:
```
"Найди ВСЕ проблемы в этом плане: [план].
Будь конкретным: называй файлы, таблицы, эндпоинты, сценарии.
Каждая проблема: что сломается, когда, почему.
НЕ говори 'выглядит нормально' — если не нашёл — смотри хуже."
```

Использовать роутинг из RECURSIVE-CRITICISM.md (какие персоны для какого типа задачи).
Получить критику. НЕ начинать выполнение.

→ БЕЗ ПОЛУЧЕННОЙ КРИТИКИ — шаг 4 не начинается.

---

### ШАГ 4 — МОЙ ОТВЕТ НА КРИТИКУ (артефакт: response-table)

Формат — таблица, строка на каждый пункт критики:

| # | Пункт критики | Принято/Отклонено | Изменение или причина отклонения |
|---|--------------|-------------------|----------------------------------|

**Запрещённые ответы:** "учёл", "принято", "хорошее замечание" — без конкретного изменения.
Каждый пункт должен иметь строку. Пропустить нельзя.

→ БЕЗ ПОЛНОЙ ТАБЛИЦЫ — шаг 5 не начинается.

---

### ШАГ 5 — РОЙ КОНТРКРИТИКУЕТ (артефакт: critique-round-2.md)

Отправить рою: обновлённый план + таблицу ответов.
Промпт:
```
"Найди что я пропустил в своих ответах.
Что я неправильно отклонил? Какие новые риски появились после моих изменений?
Проверь: мои ответы адресуют корень проблемы или только симптом?"
```

→ БЕЗ ВТОРОЙ КРИТИКИ — шаг 6 не начинается.

---

### ШАГ 6 — ФИНАЛЬНЫЙ ПЛАН (артефакт: final-plan.md)

Обновить план с учётом раунда 2.
Пометить каждое изменение: `[R1]` — из раунда 1, `[R2]` — из раунда 2.
Если раунд 2 нашёл Critical проблему → раунд 3 (повторить шаги 3-5).

→ ФИНАЛЬНЫЙ ПЛАН ГОТОВ — начинать выполнение.

---

### ШАГ 7 — ВЫПОЛНЕНИЕ (артефакт: коммит/изменения)

Следовать финальному плану точно.
- engineering:code-review после каждого изменения > 50 строк
- Если нужно отклониться от плана → задокументировать причину перед отклонением
- Не объявлять "готово" пока все шаги плана не выполнены

→ ВСЕ ШАГИ ПЛАНА ЗАВЕРШЕНЫ — писать отчёт.

---

### ШАГ 8 — ОТЧЁТ (артефакт: work-report.md)

```
СДЕЛАНО:    [что реализовано — конкретные файлы, функции, таблицы]
НЕ СДЕЛАНО: [что отложено и почему]
СЮРПРИЗ:    [что план не предусмотрел — честно]
РИСКИ:      [что ещё может сломаться]
```

Отправить рою на проверку (шаг 9).

---

### ШАГ 9 — РОЙ ПРОВЕРЯЕТ РАБОТУ (артефакт: work-verdict.md)

Запустить Agent (haiku):
```
"Проверь этот отчёт о работе: [отчёт].
Соответствует ли реализация заявленному?
Что пропущено? Что выглядит неправильно?
Сравни с оригинальным планом: [финальный план]."
```

Если рой находит проблемы → исправить → вернуться к шагу 8.
**Готово только когда рой не находит расхождений.**

---

### ШАГ 10 — CEO (3 строки максимум)

```
Результат: [что сделано]
Бизнес-эффект: [что изменилось для продукта]
Вопрос: [один вопрос если нужен]
```

Никогда: процесс, ошибки, промежуточные шаги, "я думал о..."

---

## Исключения (требуют явного разрешения CEO)

| Ситуация | Что можно пропустить | Что нельзя пропустить никогда |
|----------|---------------------|-------------------------------|
| Hotfix (прод сломан) | Шаги 3-6 | Шаги 0, 1, 7, 8, 9 |
| Typo fix (< 5 строк) | Шаги 3-6 | Шаги 0, 1, 7, 8, 9 |
| Любая другая задача | Ничего | Весь протокол |

"Задача кажется простой" — не исключение. CLASS 1 ошибки всегда начинаются так.
